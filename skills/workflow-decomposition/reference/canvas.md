# Workflow decomposition canvas

A one-page frame for the interview. Fill it with the user, one row at a time. It maps
directly onto a SKILL.md: the job/trigger become the description, the steps become the
body, and the good/bad columns become the example and guardrails.

```
USE CASE        : ___________________________________________
                  (one sentence — what this workflow is)

TRIGGER         : ___________________________________________
                  (the words / situation that should invoke it)

STEPS           DATA SOURCE             NOTES
1. __________   __________________      ____________________
2. __________   __________________      ____________________
3. __________   __________________      ____________________
4. __________   __________________      ____________________

GOOD OUTPUT     : ___________________________________________
                  (what a great result looks like — be specific)

BAD OUTPUT      : ___________________________________________
                  (failures and never-dos — the guardrails)

DEFINITION OF DONE :
                  ___________________________________________
```

## Tips for the facilitator / agent

- If the user can't name a data source for a step, that step may be doing two things —
  split it.
- "Good output" is easiest to capture by pointing at one real past example.
- The bigger the gap between good and bad, the more the guardrails matter in the skill.
- Stop when the canvas is true, not when it's full. Empty rows are fine.
