Package It as a Plugin

The clean split: individuals use skills and connectors; a plugin is for teams. It bundles those skills, connectors, agents, hooks, and commands into one install you hand to a group of people. That's why it sits in the governance session — a plugin is the unit of governed distribution, the thing an admin can require, restrict, or withhold by group.

Govern & Roll Out · ~15 min
01 — What a plugin bundles

Everything a workflow needs, in one install

A skill is a single capability. A plugin is the package around it: any mix of skills, MCP connectors, sub-agents, slash commands, and hooks, installed together.

What's inside

The skills hold the know-how, connectors bring in the data, sub-agents and slash commands carry out actions, and hooks handle automation. One install instead of a dozen setup steps.

Who actually builds them

Rarely an individual. Sometimes a department lead packaging their team's way of working. Most often the enterprise, saying "these are the tools you should be using — here's the plugin." That's the shape to plan for, and it's why this lesson is aimed at admins rather than end users.

Where it runs

Plugins run in Cowork and Claude Code, though not in Chat (bundled skills still work there). Hooks and sub-agents run only in Cowork. The layout puts plugin.json in .claude-plugin/ and skills/, agents/, and hooks/ at the root.

02 — Marketplaces

Install and adapt instead of starting blank

Anthropic ships built-in marketplaces, and you can add your own from GitHub. For our clients, the industry packs are the head start.

Built-in packs

Knowledge Work, Finance, Legal, Life Sciences

A Knowledge Work marketplace is there by default, along with Financial Services, Legal, and Life Sciences. You can add any GitHub repo as a marketplace by owner/repo or full URL.

Plugins are just markdown files, so they're easy to read and fork.
Portable by design

Author once in the Claude Code structure

Build in the Claude Code plugin structure — it's the superset, and the same plugin runs in Claude Cowork and Claude Code with nothing to rebuild. Keep the SKILL.md files tool-agnostic so they stay portable.

One source of truth; a fix ships to everyone at once.
🏛️
Prerequisite — both switches, not one

Cowork and Skills must both be enabled for the organization before you can manage plugin marketplaces at all. If the Plugins area looks empty or unavailable, this is almost always why.

03 — The admin click-path

Adding a plugin, step by step

Everything below lives at Organization settings → Plugins. Three ways in, depending on where the plugin comes from. Verified against Anthropic's admin documentation, 22 July 2026.

01
Anthropic's own marketplaces
Add plugins → Browse Anthropic sources → Add on each marketplace you want. Knowledge Work is already added by default — if you don't want it, open its menu (upper right) and choose Remove.
02
Upload your own — quick iteration
Add plugins → Upload a file → Upload to a new marketplace (first time only), name the marketplace, then drag in .zip files and click Upload. Best while you're still refining. Uploading a plugin with an existing name overwrites the previous version automatically — that's your update mechanism, and your accident.
03
GitHub sync — plugins as code
Add plugin → GitHub, then enter the repo as owner/repo. The repo must be private or internal — public repos are rejected — and the Claude GitHub App must be installed. Then open the marketplace menu and toggle Sync automatically. Syncs fire when a PR with a version bump merges to the default branch; direct pushes don't trigger a sync.
Limits to plan around

50 MB per plugin ZIP · 100 plugins per manually-uploaded marketplace but 500 via GitHub sync · plugin names max 64 characters, lowercase-with-hyphens (deployment-tools) · 30-minute sync timeout.

Then decide who gets it

Set an install preference per plugin, and on Enterprise use the Custom access column → Add groups to override it per group. Changes land on each member's next session or plugin refresh — not instantly. That's the access question, and it's the next lesson.

04 — What's inside the folder

The plugin layout

You don't need to author one to govern one, but you do need to recognise a well-formed plugin when someone hands you a ZIP. Verified against Anthropic's plugins reference, 22 July 2026.

enterprise-plugin/
├── .claude-plugin/
│   └── plugin.json        # the manifest — the ONLY thing that lives here
├── skills/
│   ├── code-reviewer/
│   │   └── SKILL.md
│   └── pdf-processor/
│       ├── SKILL.md
│       └── scripts/
├── commands/              # skills as flat .md files
├── agents/                # subagent definitions (.md)
├── hooks/hooks.json       # JSON, not markdown
├── .mcp.json              # MCP servers
├── .lsp.json              # language servers
├── monitors/monitors.json # background monitors
├── output-styles/  themes/  bin/  scripts/
└── settings.json
The mistake everyone makes

.claude-plugin/ holds only plugin.json. Every other directory — skills/, agents/, hooks/, commands/ — must sit at the plugin root, not inside it. Nesting them there is the single most common reason a plugin doesn't load.

The manifest is optional

Omit plugin.json and Claude auto-discovers components in the default locations and takes the name from the directory. A plugin with a single SKILL.md at its root and no skills/ folder loads as a one-skill plugin. Set version when you do use a manifest — without it, every commit counts as a new version.

Not everything is Markdown

Skills, commands and agents are .md. Hooks, MCP servers, LSP servers, monitors and themes are JSON. If a submission puts hook logic in a markdown file, it won't run.

🔄
Two gotchas that matter when you push an update

Changes don't all land at once. An edit to a skill's SKILL.md takes effect immediately in the current session, but changes to hooks/, .mcp.json, agents/ and output-styles/ do not — members need /reload-plugins or a restart. Say so when you push a control change, or you'll assume a hook is live when it isn't.

A CLAUDE.md at the plugin root is ignored. It is not loaded as context. If you want instructions to reach Claude, they belong in a skill — this catches people who assume the plugin can carry its own standing instructions.

05 — Design plugins to be trusted

Bundle tightly and watch what it can touch

Every connector and hook widens what a plugin can do, which is why restraint pays off before you push it to a team.

01
Scope to one job
A focused plugin for one workflow beats a sprawling one. Fewer, well-chosen parts are easier to trust, govern, and keep current.
02
Watch connectors and hooks
Connectors reach data and hooks automate actions. Stick to least privilege, gate the writes, and review what each part can do before it ships.
03
Keep skills tool-agnostic
So the same plugin runs in Claude Cowork and Claude Code without a rewrite, and stays simple to maintain.
🧩
Build a library you can reuse

Nimble Gravity's repeatable asset is a library of portable, well-scoped plugins by industry. Start each one in the Claude Code structure, keep the SKILL.md files tool-agnostic, and version them so a fix reaches every team at once.
